# Design principles

How explorations in this repository are meant to be written and read.

---

### Clarity before complexity

Explain from the simplest meaningful form before adding abstraction.

### Structure before syntax

The structure behind the notation matters more than the notation itself. Notation should reveal the idea, not hide it.

### Multiple views are better than one

When possible, include symbolic, visual, computational, geometric, discrete, categorical, or structural views of the same idea.

### Invariants matter

When perspective, representation, or scale changes, ask:

> What stays the same?

### Composition is fundamental

Small structures should read as parts of larger systems. Good abstractions compose.

### Visual thinking is part of reasoning

Diagrams, graphs, plots, and visual models are tools for thought, not decoration.

### Computation can clarify theory

Code and notebooks test, visualize, and interact with ideas — often revealing structure that static text alone does not show.

See also [CORPUS.md](CORPUS.md) for how papers (`F-*`) and notebooks (`F-*` / `Fd-*`) fit the corpus.
